# Separation of Duties

- No single person should have enough access or control to complete a critical process from start to finish on their own
- SoD divides responsibilities among multiple independent roles
- Separation of Duties is enforced through access controls, workflows, and role definitions
- Shared accounts and excessive privileges violate SoD principles

### Why Separation of Duties matters

- Insider threats
- Fraud
- privilege abuse
- Human error
- single point of failure

### Separation of Duties Model

1. Initiate ( Request / Create )

    - What this role does
        - Starts the process
        - Submits a Request
        - Creates a transaction, change, or record
    - _Examples_ 
        - Employee requests system access
        - Developer submits a code change
        - Finance staff creates a payment request

2. Approve ( Authorize / Validate )

    - What this role does
        - Reviews the request
        - Confirms legitimacy and business need
        - Authorizes or rejects the request
    - _Examples_
        - Manager approves user access
        - Finance manager approves payment

3. Execute / Review ( Perform / Implement / Verify )
    
    - What this role does
        - Carries out the approved action
    - _Examples_
        - IT grants system access
        - Admin deploys approved change
        - Finance releases payment

4. Review ( Audit)
    
    - What this roles does
        - Verifies actions after execution
        - Reviews logs, reports, and outcomes
    - _Examples_
        - Security reviews access logs
        - Audit reconciles financial transactions
        - Compliance checks change records
