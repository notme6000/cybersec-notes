# Need to Know and Least Privilege

## 1. Need to Know

- A user, system, or process is given access only to the information required to perform its job and nothing more.

### Why Need-to-Know Exists

- Limits damage if an account is compromised
- Reduces insider threats
- Protects sensitive data
- Supports compliance and audits
    - ISO 27001
    - NIST SP 800-53
    - HIPPA

### How Need-to-Know is implemented

1. Access Control 
    - Role-Based Access Control ( RBAC )
    - Attribute-Based Access Control ( ABAC )

2. Data Classification
    - Public
    - Internal
    - Confidential
    - Restricted / Secret
3. Segmentation
    - Network Segmentation
    - Database separation
    - File system permissions

### Example ( Real-World )

- HR employee :
    - _Access_ : employee records
    - _No Access_ : financial systems, source code
- IT admin :
    - _Access_ : System configuration
    - _No Access_ : payroll data 


## 2. Least Privilege

- A user, application, or system is granted the minimum permissions and access rights needed to perform its task for the minimum amount of time.

### Why Least Privilege is Critical

- Limits damage from compromised accounts
- Prevents Privilege escalation
- Reduces insider misuse
- Shrinks attack surface
- Required by nearly all compliance framework
    - NIST (SP 800-53-AC & PL controls)
    - PCI DSS

### Where Least Privilege is applied

- _Users_ :  
    - Standard users != admin
    - Admin rights only when necessary
- _Applications and Services_ :
    - Service accounts have only required permissions
    - no "god-mod" service accounts
- _Systems and OS_ :  
    - File permissions (read/write/execute)
    - Registry and system settings
- _Cloud and APIs_ : 
    - IAM roles with scoped permissions
        - Identity and Access Management  
    - Token-based, short-lived access

### Real-World Examples

- Developer can deploy code, but cannot modify payroll
- Helpdesk can reset passwords, but cannot access databases
- Script can read logs, but cannot delete them
